# PLAN — Browser Automation Latency Enhancements

## Goal

Reduce end-to-end latency for the agent's web research, content extraction, and UI verification workflows **without weakening the existing browser-skill boundaries** between `chrome-devtools-mcp`, `playwright-mcp`, and `agent-browser`. Achieve this primarily by (a) making the cheapest viable tool the obvious default, (b) shrinking per-action tool-call counts, and (c) adding two new tiers (HTTP-only and discovery) below the current 1.5–6 s browser band so trivial work does not pay browser cost.

## Background

- The current three browser skills are well-differentiated by **target**: `chrome-devtools-mcp` for Chrome DevTools-context debugging, `playwright-mcp` for Firefox/WebKit/Edge cross-engine work, `agent-browser` for routine Chromium automation. They are not differentiated by **latency tier**.
- All three skills sit in the 1.5–6 s per-task band even for trivial read-only work. A typical "read this article" task that needs no JavaScript still spawns a browser.
- `agent-browser` already supports two underused latency wins:
  - `--engine lightpanda` — CDP-compatible headless engine, ~10× faster cold start and ~10× less memory than Chrome, no GPU. Mentioned in `references/advanced.md` but not surfaced as a decision rule in `SKILL.md`.
  - `--session <name>` parallel isolated sessions and `batch` multi-command-per-process — documented but not framed as the multi-source research pattern.
- `agent-browser`'s `--auto-connect state save ./auth.json` plus state replay is the strongest free anti-bot path *and* the strongest latency path for authenticated sites: solve Cloudflare/login once in the user's real Chrome, then replay every subsequent fetch as plain HTTP. Today this is documented as an authentication convenience, not a latency strategy.
- Repo-managed `playwright-mcp` and `chrome-devtools-mcp` defaults are now headless with `--caps=testing` and slim mode where appropriate. Per-action overhead is the dominant remaining latency cost for those skills.
- 2026 research (Databay's CreepJS/browserleaks matrix, Roundproxies Patchright alternatives, Rebrowser CDP-detection deep dive, Capsolver Cloudflare 2026 methods, Spider.cloud cross-WAF guide) confirms that:
  - Patch-based stealth libraries (`puppeteer-extra-stealth`, `playwright-stealth`) often *increase* detectability on the stealth axis. Architecture (engine swap or CDP-direct) wins, not patches.
  - The `Runtime.consoleAPICalled` CDP leak (DataDome, June 2024) is now the dominant tier-3 detection vector. Patchright and nodriver fix it by skipping `Runtime.Enable`.
  - Auth-state harvesting from a real human session beats every synthetic stealth approach for both detection and latency.
- A self-hosted SearXNG is already running in the user's environment but is not wired into any harness as an MCP server. It solves the search-engine-throttling problem but does not solve the page-fetch problem; that distinction must be encoded in the skill description so the agent does not misroute fetches through SearXNG.
- The repo intentionally does **not** configure a Playwright Chromium MCP server, keeping `agent-browser` as the Chromium default. This plan preserves that decision.
- The repo's eval framework lives at `evals/skill-loading/`. There is no latency eval today; future regressions can silently undo the wins this plan targets.

## Acceptance Criteria

- A typical read-only research task ("read this article and summarize") completes through an HTTP-only path (~50–200 ms) when the page is server-rendered, without touching a browser. Verified by latency eval.
- A typical JS-rendered read task uses Lightpanda (~300–700 ms) by default and only escalates to Chrome when Chromium fidelity is required. Verified by latency eval and routing eval.
- A multi-source research task (5–10 URLs) runs in parallel via `agent-browser --session` + Lightpanda or via parallel `http-fetch` calls, not serially. Documented and demonstrated in skill recipes.
- The auth-state harvesting workflow (`--auto-connect → state save → http-fetch with cookie jar`) is documented as the recommended primary path for any authenticated, low-latency fetch. Cross-linked between `agent-browser` and the new `http-fetch` skill.
- A new `http-fetch` skill is installed via `setup.sh` and exposes at minimum: `fetch`, `fetch_json`, `fetch_md` (HTML→markdown), `fetch_concurrent`, with on-disk cache and `curl_cffi` TLS impersonation. Self-contained per AGENTS.md "iron rule" — no references outside the skill directory.
- A new SearXNG MCP integration is opt-in via `setup.sh ... --with-searxng-mcp --searxng-url=<url>`. Wires into OpenCode, Codex, Copilot, and Kiro using the same per-integration patterns as `--with-context-mode`, `--with-playwright-mcp`, and `--with-chrome-devtools-mcp`. Added to `--with-all-integrations`.
- A new `web-research` orchestrator skill exists and encodes the latency-first decision tree (SearXNG → http-fetch → Lightpanda → Chrome → fail-with-explanation). Routing evals prove it is selected for "research X" / "find docs on Y" / "what's the latest on Z" intents.
- All three existing browser skills have updated SKILL.md headers that surface the latency-relevant rules: Lightpanda decision rule (agent-browser), default `domcontentloaded` (all three), `includeSnapshot: true` as a hard rule for action tools (chrome-devtools-mcp, playwright-mcp), `evaluate_script` / `browser_run_code_unsafe` over snapshot+parse for boolean state checks (chrome-devtools-mcp, playwright-mcp).
- Skill-loading evals continue to enforce the existing routing boundaries (`chrome-devtools-mcp` vs `agent-browser` vs `playwright-mcp`) and add new boundary cases for `http-fetch`, `searxng`, and `web-research`.
- A latency eval suite exists under `evals/latency/` covering at least 10 read-only public URLs, 5 JS-rendered URLs, and 3 known-Cloudflare URLs. Reports p50 and p95 per path. Runs in `scripts/test-battery.sh` or a dedicated runner.
- All new `references/*.md` links resolve from the installed symlinked skill directory. `scripts/test-battery.sh` passes.

## Decisions

- 2026-05-20 — **HTTP-only `http-fetch` is the largest single latency win and ships before any new browser-tier skill.** — Most agent web work is read-only and the browser tax is 10–100×. This skill captures that workload at <200 ms.
- 2026-05-20 — **Lightpanda is the default engine for read-only JS-rendered tasks in `agent-browser`.** — Already supported; surface as a SKILL-level decision rule rather than buried in `references/advanced.md`. Falls back to Chrome for any task needing extensions, profiles, saved state, or WebGL/canvas/audio fidelity.
- 2026-05-20 — **Auth-state harvesting is the recommended low-latency authenticated path.** — Solves Cloudflare/login *once* in the user's real Chrome via `--auto-connect`, then replays as plain HTTP. Documented as the primary path in both `agent-browser` and the new `http-fetch` skill.
- 2026-05-20 — **SearXNG is wired as a discovery-only MCP integration.** — Returns URLs and snippets; never fetches destination pages. Hand-off to `http-fetch` (cheap) or `agent-browser` (when JS is needed). Skill description encodes this contract so the agent does not misroute fetches through SearXNG.
- 2026-05-20 — **A `web-research` orchestrator skill encodes the latency-first decision tree.** — The agent should not have to triage 5–6 tools manually for research intents. The orchestrator skill provides a single recommended entry point for "research X" / "find docs on Y" intents.
- 2026-05-20 — **Do not add a stealth-browser skill (Patchright / nodriver / Camoufox) in this plan.** — Each adds 3–5 s of cold-start latency. They are tier-3 tools for when blocked, not for routine work. Auth-state harvesting + http-fetch sidesteps most cases that would have needed them. A separate plan can revisit if real evidence shows tier-3 WAF blocking the agent during normal research.
- 2026-05-20 — **Do not add a CAPTCHA solver integration.** — Round-trip is 5–30 s. Wrong answer for a low-latency interactive coding agent. Right answer for a different repo / production scraping pipeline.
- 2026-05-20 — **Do not add a Chromium Playwright MCP server.** — Would be a slower duplicate of `agent-browser`. The repo's existing decision to keep `agent-browser` as the Chromium default holds.
- 2026-05-20 — **Existing browser skills are edited, not replaced.** — The three-skill contract (chrome-devtools-mcp / playwright-mcp / agent-browser) is well-scoped. SKILL.md edits surface latency rules; the skills' purposes and boundaries remain unchanged.
- 2026-05-20 — **Use `mcp-searxng-enhanced` (OvertliDS) as the default SearXNG MCP server.** — Most feature-complete, category-aware, supports web-scraping and date/time. Lean enough on context budget. Alternatives (`searxng-mcp` PyPI, `mcp-searxng` Node, `one-search-mcp`) noted in docs for users who prefer them.
- 2026-05-20 — **Latency evals are mandatory.** — Without them, future SKILL.md edits or upstream changes can silently regress the wins. Eval is opt-in in `test-battery.sh` (long-running) but tracked under `evals/latency/`.

## Discoveries

- 2026-05-20 — `agent-browser --engine lightpanda` is supported today and is roughly 10× faster than Chrome cold start, but the SKILL.md does not surface it as a decision rule. Estimated 5–10× latency reduction for read-only JS-rendered tasks with one SKILL.md edit and zero new dependencies.
- 2026-05-20 — `agent-browser` already has `--session <name>` (parallel isolated sessions) and `batch` (multi-command-per-process) for high-throughput multi-source work; combined with Lightpanda, a 5-source serial 15 s research task drops to ~400 ms parallel. The pattern is documented but buried under "Common Patterns".
- 2026-05-20 — The `Runtime.consoleAPICalled` CDP leak (DataDome, June 2024) is now the dominant tier-3 detection vector; standard Puppeteer/Playwright/Selenium drivers trigger it on every page load. `agent-browser` is CDP-direct and structurally similar to nodriver/Patchright on the leak axis, so its detection profile is already better than vanilla `playwright-mcp` for similar work.
- 2026-05-20 — Databay's CreepJS/browserleaks matrix (2026-05-06) shows that `puppeteer-extra-stealth` raises stealth-tool-detection from 0% to 80% relative to vanilla Puppeteer — patches advertise themselves through prototype-chain `[native code]` mismatches. Confirms the decision to skip a stealth-browser skill.
- 2026-05-20 — Capsolver / Browserbase / Browserless / Bright Data Web Unlocker are managed bypass services that round-trip in 5–30 s; they belong in a future production-scraping plan, not this latency-focused one.
- 2026-05-20 — SearXNG instances are aggressively rate-limited by Google in 2026 (issue searxng/searxng#5867, public Stack Overflow threads). Per-engine residential proxy for Google in `settings.yml` is the standard mitigation. Brave Search (with API key) and Bing remain looser fallbacks. The integration plan must not assume Google works without operational care; the `searxng` skill description must not promise comprehensive Google coverage.
- 2026-05-20 — `setup.sh` already has a clean per-integration pattern (`--with-context-mode`, `--with-playwright-mcp`, `--with-chrome-devtools-mcp`, `--with-all-integrations`); SearXNG fits naturally as a fourth `--with-searxng-mcp` flag with the same wiring across OpenCode / Codex / Copilot / Kiro.
- 2026-05-20 — `scripts/context-mode-config.py` is the OpenCode managed-overlay generator; the SearXNG integration must update it (not just `setup.sh`) so OpenCode emits the SearXNG MCP server in its managed `opencode.json`.
- 2026-05-20 — AGENTS.md "iron rule" requires self-contained skills (no references outside the skill directory). The new `http-fetch` skill must bundle its Python entry point inside `skills/http-fetch/scripts/`. Invocation via `uvx` or `pipx run` keeps it self-contained without forcing global installs.
- 2026-05-20 — `web_fetch` (the existing tool surface) is single-shot, has no TLS impersonation, no cache, no JSON-aware extraction, and no markdown conversion. The new `http-fetch` skill is the right place to fix all of those at once for the agent's read workload.
- 2026-05-20 — The biggest free anti-bot win is also the biggest latency win: solve Cloudflare/login once via `agent-browser --auto-connect state save`, then replay as HTTP. This pattern is currently framed as authentication convenience and needs to be reframed as the recommended primary path for any authenticated low-latency fetch.
